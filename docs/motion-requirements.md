# MotionMagic AI Gesture Recognition Requirements

이 문서는 **MotionMagic** 프로젝트의 AI 제스처 인식 모듈의 사양 및 요구사항을 정의합니다. 다음 단계의 고급 AI 에이전트가 `/grill-me` 인터뷰를 통해 설계를 정교화할 수 있도록 아키텍처, 제스처 사양, 데이터 파이프라인 및 핵심 인터뷰 질문들을 정리해 두었습니다.

---

## 1. 프로젝트 아키텍처 개요
MotionMagic은 실시간 웹캠 입력을 처리하는 **AI 모듈**, 이를 게임과 연결하는 **브릿지 모듈**, 그리고 게임 비주얼과 플레이 규칙을 실행하는 **게임 모듈**로 나뉩니다.

```mermaid
graph LR
    subgraph AI Module (src/ai)
        Collector --> Preprocessor
        Preprocessor --> Dataset
        Dataset --> Model
        Model --> Trainer
        Recognizer
        RuleValidator
        AimTracker
    end

    subgraph Bridge Module (src/bridge)
        CameraThread
        GestureEvent
    end

    subgraph Game Module (src/game)
        GameLoop
    end

    CameraThread --> Recognizer
    CameraThread --> AimTracker
    Recognizer --> RuleValidator
    RuleValidator --> GestureEvent
    AimTracker --> GestureEvent
    GestureEvent --> GameLoop
```

> [!IMPORTANT]
> **의존성 단방향 제약:**
> AI 모듈(`src/ai/`) 내의 그 어떤 코드도 `src/game/`이나 `pygame` 라이브러리를 직접 가져와서(Import) 사용해서는 안 됩니다. 모든 데이터 전달은 `src/bridge/` 인터페이스를 거칩니다.

## 2. AI 하위 모듈별 요구 사양

### ① collector.py (데이터 수집기)
* **역할:** 카메라 프레임에서 MediaPipe Hand Landmarker를 구동하여 손동작 학습 데이터를 생성합니다.
* **요구사항:**
  * 프레임별 **21개 손가락 관절의 3D 좌표(x, y, z)**를 획득합니다.
  * 데이터 라벨링(예: `0: rock`, `1: paper` 등)을 위한 간단한 CLI 입력을 지원해야 합니다.
  * 수집된 데이터는 `data/raw/` 디렉터리에 타임스탬프 기반 파일(JSON 또는 CSV)로 저장되어야 합니다.

### ② preprocessor.py (데이터 전처리 및 정규화)
* **역할:** 손의 크기, 거리, 손목 위치 변화에 무관하게 제스처를 인식할 수 있도록 데이터를 보정합니다.
* **요구사항:**
  * **위치 불변성 (Translation Invariance):** 손목 좌표(Landmark 0)를 기준으로 모든 좌표를 평행 이동(Zero-centering)합니다.
  * **크기 불변성 (Scale Invariance):** 손목부터 중지 끝 혹은 손바닥 중심의 거리를 기준으로 전체 좌표 스케일을 정규화(Normalization)합니다.
  * **회전 보정 (Rotation Normalization, 옵션):** 손의 기울어짐에 대응하기 위한 회전 정렬 방식 결정이 필요합니다.

### ③ dataset.py (PyTorch 데이터셋)
* **역할:** 전처리된 데이터를 PyTorch 모델 학습용 `Dataset` 및 `DataLoader`로 포장합니다.
* **요구사항:**
  * `data/processed/` 내 전처리된 데이터를 훈련(Train), 검증(Validation), 테스트(Test) 셋으로 분할(Split)하여 로드합니다.
  * 과적합(Overfitting) 방지를 위한 데이터 증강(Data Augmentation, 예: 약간의 랜덤 노이즈 추가, 좌우 반전 등)을 지원합니다.

### ④ model.py (제스처 분류 모델)
* **역할:** 손가락 관절 데이터를 기반으로 제스처 클래스를 분류하는 PyTorch 신경망을 정의합니다.
* **요구사항:**
  * **입력 차원:** `(B, 21, 3)` (또는 전처리 방식에 따라 flat한 `(B, 63)`)
  * **모델 아키텍처:** 다층 퍼셉트론(MLP) 혹은 1D Convolutional Neural Network(1D CNN).
  * **출력 차원:** 정의된 제스처 클래스 개수만큼의 소프트맥스(Softmax) 확률 분포.

### ⑤ trainer.py (모델 학습 및 검증)
* **역할:** 수집된 데이터셋으로 모델을 학습시키고 성능 지표를 도출합니다.
* **요구사항:**
  * Loss Function, Optimizer, Learning Rate Scheduler를 유연하게 구성할 수 있어야 합니다.
  * 학습 과정의 Loss 및 Accuracy 변화를 기록하고 그래프(matplotlib)로 저장합니다.
  * 최적 성능을 보인 에포크(Epoch)의 모델을 `models/` 디렉터리에 `.pth` 가중치 파일로 내보냅니다.

### ⑥ recognizer.py (실시간 인식기)
* **역할:** 학습된 `.pth` 모델을 로드하여 백그라운드 카메라 프레임 스트림으로부터 실시간으로 제스처를 예측합니다.
* **요구사항:**
  * 실시간 추론 시 지연 시간(Inference Latency)이 **15ms 이하**여야 합니다.
  * 추론 결과의 신뢰도(Confidence score)가 임계값(Threshold, 예: 85%) 이하인 경우 오작동 방지를 위해 `None`을 리턴해야 합니다.

### ⑦ rule_validator.py (규칙 기반 오탐 검증기)
* **역할:** 신경망(CNN/MLP) 모델의 오탐을 한 번 더 걸러주기 위해 기하학적 뼈대 규칙을 활용해 2차 검증을 수행합니다.
* **요구사항:**
  * 각 손가락의 굽힘/폄 상태(Fingers Extended State)를 기하학적 각도나 거리를 기준으로 분석합니다.
  * 예: '바위(Rock)' 제스처로 분류되었으나 엄지/검지 손가락 중 하나라도 펴져 있는 경우, 검증 실패 처리하여 취소합니다.

### ⑧ aim_tracker.py (조준선 추적기)
* **역할:** 손의 특정 랜드마크 좌표를 게임 해상도(1920x1080)에 대응하는 조준점 좌표로 정밀 맵핑합니다.
* **요구사항:**
  * 검지 손가락 끝(Landmark 8) 또는 손바닥 중심의 화면 맵핑 공식을 가집니다.
  * 손떨림이나 프레임 드랍으로 인한 크로스헤어 흔들림을 방지하기 위해 **Smoothing 필터 (지수 이동 평균, 1차 저주파 통과 필터, 또는 칼만 필터)**를 구현해야 합니다.

## 3. 제스처 목록 및 시나리오 사양

| 제스처 분류 | 매칭 액션 / 마법 | 판정 조건 설명 및 역할 |
|---|---|---|
| **가위 (Scissors)** | 스택용 제스처 | 검지, 중지만 펴진 상태. 입력 시 제스처 스택에 저장. |
| **바위 (Rock)** | 스택용 제스처 | 모든 손가락이 완전히 접힌 상태. 입력 시 제스처 스택에 저장. |
| **보 (Paper)** | 스택용 제스처 | 모든 손가락이 완전히 펴진 상태. 입력 시 제스처 스택에 저장. |
| **트리거 (Trigger)** | 크로스헤어 / 주문 발동 | 권총 모양. 평소엔 조준선만 이동, 손가락 스냅(발사 모션) 감지 시 스택된 제스처 조합 주문 발동. |
| **양손 합장 (Clasp)** | 마나 충전 | 두 손바닥이 가깝게 마주 보며 모인 상태. 지속형 마나 충전 활성화. |
| **손흥민 시그니처 (Sonaldo)** | 전장 전환 | 양손으로 사진을 찍는 듯한 사각형 모양(카메라 포즈)을 만든 상태. |

## 4. `/grill-me` 인터뷰 시 해결해야 할 핵심 의사결정 사항 (AI 에이전트 질문 가이드)

다음 단계의 AI 에이전트는 `/grill-me`를 통해 사용자에게 아래 질문들을 던져 구체적인 설계를 확정해야 합니다.

1. **데이터 저장 포맷:**
   * Raw 데이터 및 가공 데이터 저장 양식을 JSON과 CSV 중 무엇으로 할 것인가? (JSON의 경우 메타데이터 추가가 용이하고, CSV는 PyTorch 로더 작성이 간단함)
2. **입력 데이터의 차원 및 차원축 축소 여부:**
   * 모델의 입력으로 Z축(깊이) 정보를 포함한 3차원 좌표 `(21, 3)`을 그대로 쓸 것인가, 아니면 웹캠 2D 투영에 특화하여 Z축을 버리고 `(21, 2)`만을 활용할 것인가?
3. **조준점 트래킹 스무딩 알고리즘:**
   * 조준점의 흔들림을 제어하기 위해 단순 지수 이동 평균(EMA)을 쓸 것인가, 원-유로 필터(1€ Filter)나 칼만 필터를 탑재할 것인가?
4. **학습 하이퍼파라미터 및 검증 기법:**
   * 배치 크기(Batch Size), 기본 에포크(Epochs), 그리고 오버핏 방지를 위한 드롭아웃(Dropout) 비율은 어떻게 설정할 것인가?
5. **양손 제스처 감지 구조:**
   * 양손 제스처(예: 합장)를 처리하기 위해 MediaPipe Multi-Hand detection 결과를 어떤 구조로 스택할 것인가? (브릿지 레벨에서 결합할 것인지, AI 모델 자체가 두 손을 입력으로 받게 할 것인지)

## 5. 예외 처리 및 사용자 추가 기획

### ① 다중 손 감지 시 예외 처리 규칙

#### 1. 두 손이 감지되었으나 양손 제스처가 아닌 경우
* **판정 규칙:** 두 손의 상태가 양손 제스처(합장, 손흥민 포즈 등) 조건에 부합하지 않는다면, **Y축 좌표가 더 높은(화면에서 더 위에 위치한) 한 손**을 기준 제스처로 강제 채택합니다.
* **예시:** 가위를 든 왼손이 위쪽에 있고 주먹을 든 오른손이 아래쪽에 있다면, 왼손의 '가위'로만 판정하여 스택에 쌓습니다.

#### 2. 한 화면에 3개 이상의 손이 감지될 경우
* **배경 오인식 및 관전자 차단:** 웹캠 범위 내에 노이즈나 구경꾼의 손이 들어와 3개 이상 감지되는 경우의 처리 규칙입니다.
* **필터링:** MediaPipe 레벨에서 최대 감지 손 개수를 2개로 제한하며, 3개 이상 감지될 시 **검출 신뢰도 점수(Confidence Score)가 가장 높은 상위 2개의 손**만을 유효 입력으로 필터링하고 나머지는 전부 무시합니다.
* **우선순위 적용:** 선별된 2개의 손이 양손 제스처를 이루지 못할 시, 위의 1번 규칙에 따라 **Y축이 더 높은 한 손의 제스처만 최종 수신**합니다.

### ② 마법 발동 메커니즘 (스택 & 트리거)
* **주문 축적:** 플레이어가 한 손 제스처(가위, 바위, 보)를 수행하면 제스처 스택에 순차적으로 저장됩니다.
* **조준 및 조준점 이동:** 권총 모양(트리거) 제스처를 취하는 동안에는 조준선(크로스헤어)만 실시간으로 이동시킵니다.
* **최종 캐스팅:** 권총 모양 손 상태에서 총을 쏘는 듯한 **손가락 스냅 동작(Trigger Snap)**이 감지되면, 스택에 저장된 제스처들의 조합(예: 가위+주먹 ➡️ 낙뢰)에 대응하는 마법이 최종 발동됩니다.

### ③ 제스처 목록 요약
* **한 손 제스처 목록 (스택 적립):** 가위, 바위, 보
* **양손 제스처 목록 (특수 행동):** 합장 (지속형 마나 충전), 손흥민 시그니처 포즈 (전장 전환)
